# Test Strategy

Открывай этот файл, когда нужно выбрать правильный регрессионный тест или
объяснить, почему баг прошел мимо существующего покрытия.

## Базовые правила

- Тест должен защищать конкретный баг, а не случайно перепроверять половину
  системы.
- По возможности покажи, что сценарий падал до фикса и проходит после.
- Если сценарий слишком дорогой, flaky или разрушительный, допустим более
  узкий тест, но объясни компромисс.
- Сначала запускай минимальную проверку, потом расширяй ее только при росте
  риска.

## Почему тесты не поймали баг

Проверь одно из типовых объяснений:

- отсутствовал нужный сценарий;
- тест опирался на неверное предположение;
- моки скрывали реальное поведение интеграции;
- edge case не был покрыт;
- тест был слишком высокоуровневым и не проверял нужный контракт.

## Какой тест выбрать

| Тип бага | Предпочтительная защита |
| --- | --- |
| Логическая ошибка | Unit-тест на конкретную ветку или edge case |
| Null/Undefined | Unit-тест на пустой/частичный вход |
| Интеграция | Контрактный или узкий integration-тест на границе |
| Конфигурация | Тест конфигурации, parser/loader test, env-specific scenario |
| Состояние гонки / flaky | Детерминированный harness: fake clock, fixed seed, controlled scheduling |
| Производительность | Benchmark или threshold-test на узком участке |
| Безопасность | Тесты на валидацию, отрицательные сценарии, unsafe input |
| Память / ресурсы | Тест lifecycle/cleanup или leak-prone path |

## Практические подсказки

- Для flaky-багов сначала убери недетерминизм, потом фиксируй поведение.
- Для интеграций проверяй контракт на границе, а не только happy-path через моки.
- Для perf-багов не вводи шумный benchmark без реальной метрики или порога.
- Для security-багов делай негативные сценарии обязательной частью покрытия.
